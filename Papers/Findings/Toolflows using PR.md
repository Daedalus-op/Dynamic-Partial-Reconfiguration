- Used Partial Reconfiguration to implement incremental compilation
- Creates a dataflow graph of the operators

    - Their internals are black boxed and compiled and refined independently
- Levels of optimisation:

    - O1: Operators mapped onto a fixed size pages. These pages are interconnected via linking from Hoplite BF. Each island (page) can be independently compiled in parallel and replaced while the design is still running
    - O3: This is the final optimisation. Results similar to Vitis compilation. Monolithic compilation for best and most optimised results ^LrFXXboH

ZyCAP ^FMJIz4ns

- A PR configuration controller based on Zynq SoCs
- High throughput via soft DMA and ICAP which has a 400 MB/s throughput
- Provides drivers that can be used with Linux for managing and monitoring the PR process
- The controller executes independently from the processor ie the processor is free to execute it's own tasks instead of stalling until the reconfiguration process completes
- Uses the DRAM as a cache for storing the partial bitstreams. Uses LRU policy
- The DMA send out an interrupt to indicate completion of transfer of bitstream to the ICAP
- It uses 2 ports:
    - GP Port (AXI-Lite) : Used for transferring the addresses and size of partial bitstreams in the secondary memory to configure the soft DMA
    - HP Port (AXI-4) : Used for transferring the actual bitstreams from the DRAM to ICAP via the DMA  ^JSdGG2AJ
